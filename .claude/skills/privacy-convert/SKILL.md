# 개인정보처리방침 DOCX → JSON 변환 (편집기 연동)

## 원칙

파서/검증기의 단일 진실 공급원은 **이 프로젝트(privacy-policy-editor)** 다.
동일 로직을 UI(비개발자/법무 담당자) 와 CLI(이 스킬, 개발자/LLM 세션) 에서 공유한다.

| 담당 | 역할 |
|------|------|
| 편집기 CLI (`scripts/verify-docx.ts`) | DOCX 파싱, 섹션 경계, 표 병합, 스마트따옴표/번호 공백/헤딩 서식 자동 정상화, 이슈 위치(ref) 수집 |
| 편집기 CLI (`scripts/verify-json.ts`) | 기존 JSON 단독 검증(에디터의 `validateDoc`/`buildDocInfo` 동일 호출), 2개 이상 넘기면 **교차 언어 구조 정합**(섹션/라인 수) 리포트 |
| 스킬 (이 파일, LLM 오케스트레이터) | CLI 결과를 파싱해 사용자와 합의. "검토 필요" 항목을 건별 처리하고 JSON 을 일괄 치환. 추가로 오탈자·의미 이상 후보 제안 |

UI(에디터)와의 호환성: `verify-json.ts` 는 `App.tsx` 의 실시간 재검증(`validateDoc` / `buildDocInfo`) 과 동일 함수를 사용한다. 교차 언어 검사는 Doc 스키마만 쓰므로 UI 확장 시 그대로 재사용 가능. 현재 UI 는 단일 Doc 대상이라 "여러 파일 동시 비교" 만 스킬 전용 추가 기능.

## 실행 환경

현재 작업 디렉토리가 이 프로젝트 루트(이 `SKILL.md` 상위 3단계 — `.claude/skills/privacy-convert/..`) 라고 가정한다. `pnpm install` 선행 필수.

## 독립 실행 (자동 정리만)

### DOCX → JSON 변환
```bash
pnpm tsx scripts/verify-docx.ts <input.docx> <output.json>
```

이 한 줄만 돌려도 다음은 **자동 반영**된다 (사용자 개입 불필요):
- 스마트따옴표 `""''` → `"'`
- 섹션 번호 공백 정규화 (예: `第1 1条` → `11`)
- 문장 끝 더블 스페이스 1개로 축약
- 표 첫 행 thead 자동 승격 (휴리스틱 통과 시)
- `<w:del>/<w:delText>` 변경 추적 텍스트 `<s>` 로 보존

내보내기 시 에디터 헤딩 포맷으로 자동 일관화되는 항목(헤딩 bold 혼재)도 DOCX → JSON 에서는 `[AUTO-FIX]` 로만 표시되고 JSON 에는 영향 없음.

### JSON 단독 검증 / 교차 언어 정합
```bash
# 단일 파일 검증
pnpm tsx scripts/verify-json.ts <input.json>

# 교차 언어 정합 (첫 파일 기준으로 섹션/라인 수 비교)
pnpm tsx scripts/verify-json.ts <ko.json> <ja.json> <en.json>
```

출력 블록:
- `[INFO]`   : headers/sections/tables 개수 (파일별)
- `[REVIEW]` : 번호 스타일·도트 혼재, 스마트따옴표 잔존, 빈 섹션, 취소선, 번호 누락·중복
- `=== 교차 정합 ===` : 2개 이상 받았을 때, 기준 파일 대비 **섹션 수 / 조별 라인 수** 차이

`verify-json.ts` 는 자동 수정을 하지 않는다(DOCX 파서가 없으므로 AUTO-FIX 블록도 없음). 구조 드리프트 탐지 용도.

## 오케스트레이션 워크플로우

입력 유형에 따라 두 트랙으로 분기한다.

- **트랙 A (DOCX → JSON)**: 아래 0~5 단계. 1차 변환부터 수행.
- **트랙 B (JSON 검증·교차 언어 정합)**: `B0`~`B3` 단계. DOCX 가 없고 이미 JSON 이 존재할 때(예: 배포된 번역본 드리프트 점검, 기존 버전 재확인).

두 트랙은 배타적이지 않다. 예: 트랙 A 로 KO JSON 을 새로 만든 뒤, 기존 JA/EN 과 함께 트랙 B 의 교차 정합을 추가로 수행할 수 있다.

## 트랙 A — DOCX → JSON 변환

### 0. 입력 수집

사용자로부터 다음을 받는다:
- 입력 DOCX 경로
- 출력 JSON 경로 (사용자가 지정하지 않으면 입력과 같은 디렉토리에 `.docx` → `.json` 으로 제안)

### 1. 1차 변환 (스크립트)

```bash
pnpm tsx scripts/verify-docx.ts <input.docx> <output.json>
```

출력 블록:
- `[INFO]`      : headers/sections/tables 개수
- `[AUTO-FIX]`  : 자동 반영된 항목 (그대로 사용자에게 요약 보고)
- `[REVIEW]`    : 사용자 검토 필요 항목 + 위치 ref 목록

### 2. AUTO-FIX 요약 보고

이 블록은 **편집하지 않고** 사용자에게 한 단락으로 전달.

### 3. REVIEW 항목 건별 처리

각 이슈를 **아래 전략**으로 분기. JSON 수정 전 사용자 동의를 구한다.

#### (a) 섹션 번호 누락 / 중복

**처리**: 자동 수정 금지. 원본 DOCX 확인 요청.
- 번호 누락: 원문에 해당 조(예 N=7)가 존재하는데 파서가 놓쳤는지 확인
- 번호 중복: 섹션 분리가 필요한지, 한쪽이 하위 목록으로 들어가야 하는지 확인

사용자가 DOCX 를 수정해 재실행하는 것이 기본 경로. 예외적으로 JSON 수준에서 수기 편집이 필요하면 사용자가 명시 지시 후 진행.

#### (b) 빈 섹션

**처리**: 각 빈 섹션에 대해 사용자에게 옵션 제시.
- "의도적인 공란" → 유지
- "내용 누락" → DOCX 원문 확인 필요 / 수동 추가

#### (c) 취소선 / 삭제 표시 잔존 (`<s>...</s>`)

**처리**: ref 위치별로 현재 텍스트를 보여주고 사용자 선택:
1. 태그와 내용 모두 삭제 (원문 삭제 반영)
2. 태그만 벗기고 내용 유지 (취소의 취소)
3. 그대로 유지 (나중에 에디터에서)

선택이 전건 동일하면 일괄 처리 허용.

#### (d) 번호 스타일 혼재

**처리**:
1. REVIEW 의 스타일 분포 그대로 사용자에게 제시 (예: `1. (마침표) 26건, ①②③ 9건, ⅰ) 4건`)
2. 사용자에게 정책 결정 요청: "①②③ 으로 통일" / "1) 으로 통일" / "건별 수동" / "유지"
3. 정책 결정되면 `[REVIEW]` 의 ref 위치를 순회하며 해당 줄의 선행 마커를 치환.
   - `1. ` → `① ` 같은 매핑은 한 섹션 내 순번 기준으로 계산.
   - 치환 전 diff 를 보여 사용자 확인 후 적용.

#### (e) 도트 문자 혼재

**처리**:
1. 분포 제시 (예: `'·' 8개, '⋅' 8개, '․' 2개, '∙' 1개`)
2. 사용자에게 canonical 도트 문자 선택 요청 (또는 "유지")
3. 선택되면 소수 도트가 포함된 ref 위치에서 canonical 로 일괄 치환.
4. 치환 전후 비교 보여주고 확인 받음.

### 4. LLM 고유 — 오탈자·의미 이상 후보 제안 (자동 수정 금지)

편집기 UI 에서는 할 수 없고, 이 스킬만 제공하는 심화 리뷰 단계.

**목표**: 스크립트 검증기가 잡지 못하는 **자연어 수준의 이상**을 사용자에게 후보로 제시. 자동 수정은 절대 금지, 사용자가 건별 승인해야만 적용.

**검사 카테고리 (제안 범위)**:

| 카테고리 | 예시 |
|----------|------|
| 오탈자(orthography) | `파기하,` → `파기하며,`, `이용자자가` → `이용자가`, `Chrome :설정` → `Chrome: 설정` |
| 띄어쓰기/표기 일관성 | 같은 용어의 표기 흔들림 (`개인 정보`/`개인정보` 혼용 등) |
| 조사·어미 부자연 | 문장 중단, 조사 누락, 수동태 과다 |
| 숫자·법조문 인용 정합 | `개인정보 보호법 제OO조` 표기 패턴 내 숫자 누락 |
| 문장 경계 이상 | 두 문장이 공백만으로 붙어있거나, 문단 중간 `.` 없이 줄바꿈 |
| 이해 저해 표현 | 모호한 지시어 (`이러한 사항은...`)가 앞뒤 맥락 없이 등장 |

**절차**:

1. 섹션별로 JSON 을 읽고, LLM 이 후보를 뽑아 다음 포맷으로 나열:
   ```
   [N건 후보]
   1. 제3조 L7 · 오탈자
      원문: "개인정보 보호 법에 따라 ..."
      제안: "개인정보 보호법에 따라 ..." (띄어쓰기 오류로 판단)
   2. 제5조 L2 · 표기 일관성
      원문: "개인 정보 보호책임자"
      제안: "개인정보 보호책임자" (문서 내 다수 표기 기준)
   ...
   ```
2. 사용자에게 건별 `y/n/skip` 또는 "모두 수용/모두 보류" 요청.
3. **자동 수정 금지**: y 응답 확인 후에만 JSON 치환. 근거(동일 섹션 내 다수 표기 등) 를 함께 남긴다.
4. 판단이 불확실한 건은 `(불확실)` 태그로 표시하고 기본값 `skip`.

**보수적 원칙**:
- 법률 조문·기관명·외래어 표기는 손대지 않는다 (예: "개인정보보호위원회" 는 표준명, 임의 변경 금지).
- 전문용어 번역(한↔영↔일) 은 제안만, 자동 적용 금지.
- 여러 대안이 가능하면 2~3개 옵션 모두 제시.

### 5. 재확인 / 저장

수정을 반영한 최종 JSON 을 저장하고, 잔여 REVIEW 항목이 있는지 한 번 더 출력한다.
기존 버전 JSON 이 있으면 최종 diff 로 의미 차이만 남았는지 확인:

```bash
diff <(python3 -m json.tool <prev.json>) <(python3 -m json.tool <output.json>) | head -100
```

## 트랙 B — JSON 검증·교차 언어 정합

### B0. 입력 수집

사용자로부터 다음을 받는다:
- 1개 이상의 JSON 경로. 2개 이상이면 첫 파일이 **기준(baseline)** 이 된다.
- 기준을 명시(예: "KO 기준으로 JA/EN 비교") 하지 않으면, 인자 순서대로 `[base, cmp1, cmp2, ...]` 로 간주.

### B1. 검증 실행 (스크립트)

```bash
pnpm tsx scripts/verify-json.ts <base.json> [<cmp1.json> <cmp2.json> ...]
```

출력 블록:
- 파일별 `[INFO]`, `[REVIEW]`
- 2개 이상일 때 말미에 `=== 교차 정합 ===` (섹션 수 / 조별 라인 수 차이)

### B2. 결과 해석 및 건별 처리

#### (a) 파일별 REVIEW
트랙 A 단계 3 의 (a)~(e) 와 동일한 기준으로 처리. 단 `verify-json.ts` 는 DOCX 원문이 없으므로 "스크립트 자동 정리(AUTO-FIX)" 는 존재하지 않고, 수정은 모두 LLM+사용자 합의로 JSON 에 직접 반영.

#### (b) 섹션 수 불일치
**처리**: 자동 수정 금지. 원인 가능성:
- 한쪽 언어판에 조가 빠져 있음 (누락 번역)
- 한쪽 언어판에 조가 추가됨 (분할/신설)
- 섹션 패턴이 달라 파서가 다르게 잡음 (제목 포맷 점검 필요)

사용자에게 어느 파일이 올바른 기준인지 확인 후, 정정 대상 파일을 수기 편집 가이드.

#### (c) 조별 라인 수 차이
**처리**: 구조적 드리프트 후보. 다음을 우선 점검:
- 같은 문단을 한쪽은 1줄로 병합, 다른 쪽은 여러 줄로 분할 (문장 경계 불일치)
- 표 누락/추가
- 리스트 항목 수 불일치

LLM 이 차이 나는 조만 대조해 후보를 나열하고, 사용자 합의 후 **기준 파일에 맞추는 방향** 으로 수정(혹은 기준을 다른 파일로 바꾸어 재수행).

### B3. LLM 고유 — 번역 정합 후보 제안 (자동 수정 금지)

단계 4(오탈자·의미 이상) 에 더해, 교차 언어 특유의 이슈를 후보로 제시:

| 카테고리 | 예시 |
|----------|------|
| 문장 경계 드리프트 | KO 1줄 병합 / JA·EN 2줄 분할 → 어느 쪽이 문서 다른 조의 지배 패턴과 일치하는지 평가 |
| 용어 번역 일관성 | 동일 원문이 JA/EN 에서 서로 다른 역어로 나타나는 경우 |
| 숫자·고유명사 | 조 번호, 법령명, 기관명, 담당자명/직책의 표기 일치 여부 |
| 버전·일자 | 제14조(마지막 조) 의 `V1.X` / `YYYY-MM-DD` 값이 세 파일에서 같은지 |

**절차는 단계 4 와 동일**: 후보 나열 → `y/n/skip` → 승인분만 치환. 판단 불확실분은 `(불확실)` 로 표시하고 기본 `skip`. 법률 조문·공식 기관명·고유명사는 보수적으로.

## 예외 처리 가이드

| 상황 | 행동 |
|------|------|
| `pnpm install` 미실행 | 사용자에게 `pnpm install` 실행 요청 |
| `[INFO] sections: 0` | 섹션 패턴 매칭 실패. DOCX/JSON 을 열어 조 제목 형태 확인 후 사용자에게 보고 |
| `[REVIEW]` 항목 0 + `[AUTO-FIX]` 만 있음 | 스크립트 변환만으로 완료. 단계 4(오탈자 리뷰) 로 진행할지 사용자에게 질문 |
| 사용자가 "스킵" 선택 | 해당 이슈 건드리지 않음. 잔여 이슈 리스트로 다시 보고 |
| JSON `headers`/`contents` 필드 없음 | 스키마 위반. 파일이 본 에디터 포맷인지 확인 요청 |
| 교차 정합 — 기준 파일 판단 어려움 | 사용자에게 기준 파일 지정 요청. 기본값은 KO(원문) |

## 주의: 건드리지 말 것

- 도트 문자 5종(`·⋅․∙・`) 의 canonical 선택은 문서/언어별로 다를 수 있음. 자동 통일 금지. 반드시 사용자 결정.
- 번호 스타일 정책도 문서/조항별로 다를 수 있음. 일괄 변환은 사용자 명시 동의 후에만.
- 단계 4의 오탈자·의미 제안은 **제안만**. 어떤 경우에도 사용자 확인 없이 JSON 을 바꾸지 않는다.
- 법률 조문·공식 기관명·고유명사는 보수적으로 다룬다.
